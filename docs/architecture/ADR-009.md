# ADR-009: Optimistic Locking para prevenir race conditions en operaciones de cuenta

## Status
Accepted

## Context

Las operaciones financieras sobre una cuenta (`deposit`, `withdraw`) siguen este flujo:

1. Leer el estado actual de la cuenta desde la BD
2. Validar límites diarios/mensuales (SUM de transacciones completadas)
3. Validar fondos suficientes
4. Modificar el balance en memoria
5. Guardar la cuenta actualizada en la BD

Los pasos 1 y 5 no son atómicos. Dos requests concurrentes sobre la misma cuenta pueden leer el mismo estado y ambos pasar las validaciones, resultando en inconsistencias:

**Race condition en límites (Problema 1):**
```
Límite diario depósito: $200.000. Acumulado hoy: $180.000.

Hilo A: SUM=$180k → $180k + $20k = $200k ≤ límite → pasa validación
Hilo B: SUM=$180k → $180k + $20k = $200k ≤ límite → pasa validación
Hilo A: ejecuta depósito → acumulado real: $200.000 ✓
Hilo B: ejecuta depósito → acumulado real: $220.000 ✗  (límite superado)
```

**Race condition en balance (Problema 2):**
```
Balance: $100.000. Dos retiros de $80.000 simultáneos.

Hilo A: availableBalance=$100k → $100k - $80k = $20k ≥ 0 → pasa validación
Hilo B: availableBalance=$100k → $100k - $80k = $20k ≥ 0 → pasa validación
Hilo A: debita $80k → balance persiste como $20k
Hilo B: debita $80k → balance persiste como $20k
Resultado: se retiraron $160k de una cuenta con $100k.
```

El mecanismo de idempotencia existente (`IdempotencyKey`) protege únicamente contra **reintentos del mismo cliente** (doble click, red caída). No protege contra dos requests distintos y concurrentes que llegan simultáneamente con claves diferentes.

### Alternativas evaluadas

**Pessimistic locking (`SELECT FOR UPDATE`)**
Bloquea la fila al momento de leerla. El segundo request espera hasta que el primero commitee. Garantía fuerte, pero genera contención: bajo carga alta las operaciones se serializan y el throughput cae linealmente.

**Serializable isolation**
Eleva el nivel de aislamiento de toda la transacción. Muy costoso en términos de rendimiento y genera falsos positivos (conflictos entre transacciones que no compiten por el mismo recurso).

**Optimistic locking (`@Version`)**
No bloquea al leer. Cada transacción trabaja libremente. Al momento de escribir, la BD verifica atómicamente que el registro no fue modificado por otra transacción mientras tanto. Si fue modificado, el segundo request falla con una excepción y su transacción se revierte completa. El cliente recibe un 409 y puede reintentar.

Los conflictos reales sobre una misma cuenta en el mismo instante son estadísticamente raros. Optimistic locking es la elección correcta cuando los conflictos son infrecuentes: el 99.9% de las operaciones no encuentran contención y terminan sin penalización de rendimiento.

## Decision

Se implementa **optimistic locking mediante `@Version` de JPA/Hibernate** sobre la entidad `accounts`.

### Mecanismo

Se agrega una columna `version BIGINT NOT NULL DEFAULT 0` a la tabla `accounts`. Hibernate modifica automáticamente cada `UPDATE` para incluir la versión como condición:

```sql
-- Antes
UPDATE accounts SET balance = ?, updated_at = ? WHERE id = ?

-- Después (generado automáticamente por Hibernate)
UPDATE accounts SET balance = ?, updated_at = ?, version = 6 WHERE id = ? AND version = 5
```

Si `AND version = 5` no matchea ninguna fila (otra transacción ya incrementó la versión), `getUpdateCount()` devuelve 0 y Hibernate lanza `OptimisticLockingFailureException`. El `@Transactional` activo hace rollback de toda la operación, incluyendo el registro de la transacción en la tabla `transactions`. No queda ninguna inconsistencia en la BD.

La versión es incremental desde 0, nunca se resetea. `BIGINT` soporta hasta ~9.2 × 10¹⁸ escrituras por fila — el desbordamiento es imposible en la práctica.

### Flujo con dos requests concurrentes

```
Hilo A: SELECT ... version=5 ──── procesa ──── UPDATE WHERE version=5 → 1 fila ✓ COMMIT (version→6)
Hilo B: SELECT ... version=5 ────── procesa ──── UPDATE WHERE version=5 → 0 filas ✗ ROLLBACK → 409
```

Ningún lock se mantiene durante el procesamiento. El único punto de serialización es el `UPDATE` en sí, que PostgreSQL ejecuta con un lock de fila de microsegundos.

### Propagación de `version` por las capas

El campo `version` debe recorrer todas las capas para que el `UPDATE` use el valor leído originalmente:

```
BD (version=5)
  → AccountJpaEntity.version = 5     (@Version)
  → AccountJpaMapper.toDomainEntity  (copia version=5 al dominio)
  → Account.version = 5              (viaja en memoria)
  → AccountJpaMapper.toJpaEntity     (copia version=5 de vuelta a JPA)
  → Hibernate: WHERE version=5, SET version=6
```

Si `version` no se propaga correctamente al mapper de escritura, el `UPDATE` filtraría `WHERE version = null`, no matchearía ninguna fila, y la excepción se lanzaría en todas las operaciones sin concurrencia real.

## Consequences

### Positive

1. **Eliminación de race conditions** en balance y en validación de límites: ambas inconsistencias se resuelven con el mismo mecanismo porque ocurren dentro de la misma transacción que termina con `save(account)`
2. **Sin overhead en lecturas:** `SELECT` no se ve afectado; la verificación ocurre solo en `UPDATE`
3. **Sin contención entre cuentas distintas:** dos operaciones sobre cuentas diferentes nunca compiten
4. **Rollback completo ante conflicto:** `@Transactional` garantiza que el registro en `transactions` también se revierte; no hay transacciones huérfanas en estado inconsistente
5. **Cambio contenido en infraestructura:** el dominio solo transporta `version` como un `Long` opaco; la semántica de locking vive en `@Version` (JPA entity) y en el handler (infraestructura HTTP)

### Negative

1. **El cliente debe manejar 409:** operaciones legítimas pueden fallar bajo concurrencia real y requerir reintento. El frontend debe distinguir `CONCURRENT_MODIFICATION` de otros 409 para mostrar un mensaje apropiado ("La operación falló momentáneamente, intentá de nuevo")
2. **`version` cruza la barrera dominio/infraestructura:** el campo vive en el domain model aunque es un concepto de persistencia. La alternativa (manejarlo solo en la capa JPA) requeriría un mecanismo de identidad adicional entre entidad JPA y objeto de dominio que aumentaría la complejidad sin beneficio real

### Edge cases

| Escenario | Comportamiento |
|-----------|---------------|
| Mismo usuario, mismo instante, dos operaciones sobre la misma cuenta | Primera commitea, segunda recibe 409 |
| Dos usuarios con acceso a la misma cuenta (futuro) | Ídem — el mecanismo es por fila de cuenta, no por usuario |
| Operaciones sobre cuentas distintas de un mismo cliente | Sin conflicto — `version` es independiente por cuenta |
| Límites excedidos con operaciones sobre cuentas distintas (transferencias futuras) | Optimistic locking sobre la cuenta no protege este caso; requiere análisis separado |
